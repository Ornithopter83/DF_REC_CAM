# DFBlackbox Codex 작업 규칙

이 문서는 Codex가 DFBlackbox 저장소에서 작업할 때 항상 먼저 확인해야 하는 규칙이다.

## 기본 원칙

- 작업 전 관련 파일을 먼저 분석하고, 기존 구조와 동작을 이해한 뒤 수정한다.
- 큰 변경이나 여러 파일에 걸친 변경 전에는 짧은 계획을 먼저 제시한다.
- 실제 애플리케이션 기능 코드는 요청 범위 안에서만 수정한다.
- 기능 추가 또는 수정 시 기존 동작의 회귀 가능성을 확인한다.
- 작업 후 가능한 경우 빌드 또는 테스트를 실행하고, 실행하지 못한 경우 이유를 남긴다.
- 완료 보고에는 변경 파일과 변경 이유를 요약한다.
- 사용자 요청이 코드 변경이 아니라 작업 문서 정리인 경우, 실제 구현은 하지 않고 문서 범위만 수정한다.

## 변경 제한

- 기존 public API를 임의로 변경하지 않는다.
- DB 스키마를 임의로 변경하지 않는다.
- 기존 설정 파일 형식을 임의로 변경하지 않는다.
- 빌드 설정, 패키지, 배포 설정은 요청이 있을 때만 변경한다.
- WinForms `Designer.cs` 직접 수정은 최소화한다. 필요한 경우 레이아웃 영향 범위를 확인한다.
- UI 문구를 추가하거나 변경할 때는 기존 현지화 구조를 우선 사용한다.

## 보안과 로그

- 보안 정보, 비밀번호, 토큰을 코드나 문서에 기록하지 않는다.
- 전체 RTSP URL을 로그에 출력하지 않는다. 필요하면 호스트, 포트, 상태처럼 민감하지 않은 정보만 남긴다.
- 사용자 환경의 로컬 경로, 장비 주소, 인증 정보는 필요 최소한으로 다룬다.

## 리소스 관리

- 영상, 이미지, OpenCV 객체(`Mat`, `VideoCapture`, `VideoWriter` 등)는 Dispose 또는 해제 여부를 확인한다.
- 네트워크, 스트림, 파일 핸들, 프로세스 객체도 수명 관리를 확인한다.
- UI 스레드와 백그라운드 작업 간 공유 객체는 수명과 동시 접근을 함께 검토한다.
- 재생 최적화 작업에서는 디코딩, 프레임 버퍼, Bitmap/GDI 객체의 소유권과 해제 시점을 반드시 확인한다.

## 작업 기준 문서

- 새 작업 전 `docs/PROJECT_CONTEXT.md`와 `docs/DECISIONS.md`를 읽는다.
- 실제 작업은 가능하면 `tasks/*.md` 파일 단위로 나누어 수행한다.
- 작업 문서는 번호형 파일명(`00_...`, `01_...`)을 사용한다.
- 작업 완료 후 해당 task 문서의 `결과` 섹션을 최신 상태로 갱신한다.
- 긴 대화 이주용 요약을 반복 작성하지 않는다.
- 새로 결정된 사항만 `docs/DECISIONS.md`에 짧게 추가한다.

## 현재 주요 구현 방향

- UI 문구는 KOR/ENG 현지화를 고려한다.
- 동영상 재생은 파일 선택과 드래그드롭 경로가 같은 재생 흐름을 사용한다.
- 재생 최적화는 `균형 / 재생 우선 / 추적 우선` 모드를 기준으로 유지한다.
- 고FPS 재생은 불필요한 복사, 강제 repaint, UI 스레드 점유를 줄이는 방향을 우선한다.
