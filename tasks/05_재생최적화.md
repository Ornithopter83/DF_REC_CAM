# 작업명

재생 최적화

## 목표

* 1920x1080, 60fps에 가까운 고비트레이트 동영상도 가능한 한 원본 속도에 가깝게 부드럽게 재생한다.
* 현재 재생 구조에서 불필요한 UI 스레드 점유, 프레임 복사, Bitmap 변환, 강제 repaint 비용을 줄인다.
* 프레임 스킵은 최후의 보정 수단으로 두고, 우선 PC 성능을 최대한 활용하는 구조로 개선한다.
* 재생 중에도 재생 컨트롤, 타임라인, 상태바가 멈추지 않도록 UI 응답성을 유지한다.

## 배경

* 최근 확인한 테스트 영상 예시는 1920x1080, 약 54.6fps, 총 비트레이트 약 160Mbps 수준이다.
* 디코드 후 1080p BGR 프레임 1장은 약 5.9MiB이며, 54.6fps 기준 원본 프레임 데이터만 초당 약 320MiB 이상 처리된다.
* 현재 재생 루프는 프레임을 하나씩 읽고 처리한 뒤 화면에 표시하는 구조다.
* 현재 재생 경로에는 다음 비용이 누적될 수 있다.
  * `VideoCapture.Read`
  * 프레임 `Clone`
  * 최신 프레임 보관용 복사
  * 재생 오버레이용 복사 또는 그리기
  * 표시용 캔버스 생성
  * `Mat`에서 `Bitmap` 변환
  * `PictureBox.Image` 교체
  * 재생 모드에서 강제 화면 갱신
  * 상태바와 타임라인 갱신
* 한 프레임 처리 시간이 60fps 기준 예산인 약 16.7ms를 넘으면 재생 속도가 0.5x처럼 느려질 수 있다.
* 현재 문제는 단순한 하드웨어 부족보다는 재생 파이프라인이 고fps 영상에 맞게 분리/최적화되지 않은 문제로 본다.

## 변경 금지

* 기존 녹화, 카메라 연결, 감지 알고리즘의 의미를 바꾸지 않는다.
* 녹화 저장 경로, 파일 형식, 설정 파일 스키마를 임의로 변경하지 않는다.
* DB, 배포 설정, 패키지 설정은 요청 없이 변경하지 않는다.
* 전체 RTSP URL, 로컬 사용자 경로, 인증 정보가 로그에 남지 않도록 한다.
* 외부 플레이어 실행 방식으로 대체하지 않는다.
* 재생 최적화를 이유로 기존 수동 탐색, 이전/다음 프레임, 타임라인 이동 기능을 제거하지 않는다.

## 요구사항

* 재생 속도 저하 원인을 코드상 병목별로 분리해 확인한다.
* 재생 중 매 프레임 강제 repaint가 병목이면 제거하거나 제한한다.
* 재생 중 불필요한 `Clone`, 표시용 캔버스 생성, `Bitmap` 변환을 줄인다.
* 재생 오버레이가 꺼져 있을 때는 가장 가벼운 표시 경로를 사용한다.
* 재생 중 상태바와 타임라인은 매 프레임 갱신하지 말고 필요한 주기로 제한하는 방안을 검토한다.
* 디코딩과 UI 표시를 분리할 수 있는 구조를 검토한다.
* 프레임을 미리 읽는 작은 버퍼를 도입할 수 있는지 검토한다.
* 프레임 스킵은 기본 전략으로 삼지 말고, 처리 지연이 누적될 때 선택적으로 사용할 수 있는 정책으로만 검토한다.
* 재생 속도 보정은 MP4 duration 기반 FPS 계산과 기존 OpenCV 메타데이터 보정 로직을 훼손하지 않는다.
* 드래그드롭 재생, 파일 선택 재생 모두 동일한 최적화 경로를 사용해야 한다.
* KOR/ENG 현지화가 필요한 새 사용자 표시 문구가 생기면 `Localization`에 연결한다.

## 권장 구현 방향

* 1차 최적화는 구조 변경이 작은 항목부터 진행한다.
  * 재생 중 `picCameraPreview.Update()` 같은 동기 repaint 호출 제거 또는 제한
  * 오버레이가 꺼진 재생에서는 clone과 canvas 생성 최소화
  * 상태바/타임라인 갱신 주기 제한
  * `SetLatestFrame`이 재생 중 반드시 필요한지 검토하고, 필요 없으면 재생 경로에서는 생략
* 2차 최적화는 디코딩과 표시를 분리한다.
  * 백그라운드에서 프레임을 읽고 표시 준비를 수행
  * UI 스레드는 준비된 프레임을 받아 화면에 표시하는 일만 수행
  * 작은 프레임 버퍼로 디코딩 순간 지연을 흡수
* 3차 최적화는 렌더링 방식 개선을 검토한다.
  * `PictureBox.Image` 매 프레임 교체 대신 커스텀 컨트롤에서 최신 Bitmap을 그리는 방식
  * 필요 시 하드웨어 가속 렌더러 또는 영상 재생 라이브러리 검토
* 최적화 전후 비교를 위해 디버그 로그 또는 임시 계측값을 둘 수 있다.
  * 프레임 읽기 시간
  * 변환 시간
  * UI 표시 시간
  * 실측 표시 FPS
  * 재생 시간과 실제 경과 시간 차이
* 임시 계측 코드는 사용자에게 보이는 기능이 아니면 작업 완료 전 제거하거나 디버그 전용으로 제한한다.

## 결정 필요 -> 결정 완료

* 하드웨어 가속 디코딩을 사용.
* 프레임 스킵 정책을 허용할지 결정 필요.
  * 선택 : 실시간 재생 우선. 처리 지연이 누적되면 일부 프레임을 건너뛰어 실제 재생 시간과 동기화한다.
* 재생 중 오버레이 처리 범위를 결정 필요.
  * 선택 : 오버레이가 켜져 있으면 현재처럼 분석/표시를 수행한다.
* 렌더러 교체 범위 결정 필요.
  * 선택 : 현재 `PictureBox` 기반을 유지하며 최적화한다.
  * 선택 B: WinForms 커스텀 렌더링 컨트롤로 교체한다.
  * 선택 C: LibVLC, DirectShow, Media Foundation, FFmpeg 기반 별도 렌더러를 검토한다.
* 프레임 버퍼 크기 결정 필요.
  * 10프레임 버퍼 사용.
* 재생 최적화가 감지/오버레이 정확도보다 우선인지 결정 필요.
  * 배타적 옵션으로 설정 (균형 / 재생 우선 / 추적 우선)

## 완료 조건

* 컴파일 에러 없이 빌드된다.
* 기존 파일 선택 재생과 드래그드롭 재생이 모두 정상 동작한다.
* 1920x1080, 60fps에 가까운 고비트레이트 영상에서 재생 속도가 기존보다 개선된다.
* 재생 중 화면 갱신이 멈추지 않는다.
* 재생 중 UI 조작이 과도하게 버벅이지 않는다.
* 이전/다음 프레임, 재생/일시정지, 타임라인 탐색이 기존처럼 동작한다.
* 재생 오버레이 설정이 기존 의미를 유지한다.
* 불필요한 GDI/Bitmap 객체 누수 없이 이전 이미지와 프레임 리소스가 해제된다.
* 새 사용자 표시 문구가 있다면 KOR/ENG 언어 선택에 맞게 표시된다.

## 테스트 방법

* `dotnet build DFBlackbox\DFBlackbox.csproj -o .buildcheck`
* 1920x1080, 54~60fps, 고비트레이트 동영상을 파일 선택으로 재생한다.
* 같은 동영상을 드래그드롭으로 재생한다.
* 재생 속도가 실제 영상 길이와 크게 어긋나지 않는지 확인한다.
* 상태바 FPS와 체감 재생 속도를 확인한다.
* 재생 중 일시정지/재개가 정상 동작하는지 확인한다.
* 재생 중 이전/다음 프레임 이동이 정상 동작하는지 확인한다.
* 타임라인 탐색 후 재생이 정상 속도로 이어지는지 확인한다.
* 재생 오버레이를 끈 상태와 켠 상태를 나누어 비교한다.
* ENG 언어 선택 상태에서 새 사용자 표시 문구가 생겼다면 영어 표시를 확인한다.
* 장시간 재생 후 메모리 사용량이 계속 증가하지 않는지 확인한다.

## 결과

* 재생 중 오버레이가 꺼진 경우 10프레임 버퍼 기반 재생 경로를 사용하도록 추가했다.
* 버퍼 재생 경로에서는 백그라운드에서 프레임을 읽고 Bitmap으로 준비해 UI 스레드 표시 부담을 줄였다.
* 재생 중 동기 강제 repaint 호출을 제거했다.
* 오버레이가 꺼진 재생에서는 표시용 캔버스 clone을 생략하고 직접 Bitmap으로 변환하도록 최적화했다.
* 수동 탐색/프레임 이동 시 버퍼를 비우고 기존 seek 동작이 유지되도록 했다.
* OpenCV 백엔드가 지원하는 경우 하드웨어 가속 디코딩 속성을 시도하고, 실패하면 기존 소프트웨어 디코딩으로 fallback하도록 했다.
* 설정창에 재생 최적화 모드 `균형 / 재생 우선 / 추적 우선`을 추가했다.
* `균형`은 오버레이가 꺼진 일반 재생에서 버퍼 경로를 사용한다.
* `재생 우선`은 재생 속도를 우선하여 재생 오버레이를 생략하고 버퍼 경로를 사용한다.
* `추적 우선`은 기존 분석/오버레이 의미를 우선하여 기존 재생 경로를 사용한다.
* `dotnet build DFBlackbox\DFBlackbox.csproj -o .buildcheck` 빌드가 경고 0개, 오류 0개로 성공했다.
